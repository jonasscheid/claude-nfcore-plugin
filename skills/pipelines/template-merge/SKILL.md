# Template Merge

Resolve `upstream/TEMPLATE` → `dev` cleanly: keep pipeline logic, accept template infrastructure updates, and don't lose the merge mid-flight.

Read `${CLAUDE_PLUGIN_ROOT}/shared/conventions.md` first for the package-manager prefix (`<cmd_prefix>`).

---

## Branch Setup

```bash
git fetch upstream                                    # nf-core remote
git checkout -b merge-template-<X.Y.Z> upstream/dev   # nf-core uses `dev`, not `develop`
git merge upstream/TEMPLATE                           # will conflict — that's expected
```

Naming: `merge-template-<X.Y.Z>` where `X.Y.Z` is the template version (e.g. `merge-template-402` for 4.0.2). Match prior branches in the repo.

If the working tree has uncommitted changes, **commit or move them to a different branch first** — see Pitfalls.

## Per-File Conflict Resolution

Group every conflict into one of these buckets before opening the file:

| Bucket | Examples | Default |
|---|---|---|
| Pipeline logic | `workflows/`, `subworkflows/local/`, `modules/local/`, `conf/test_*.config` | **Keep ours** |
| Autogenerated metadata | `ro-crate-metadata.json` | **Keep ours** — `nf-core lint` regenerates it |
| Hybrid — read both sides | `.github/workflows/*`, `.nf-core.yml`, `nf-test.config`, `README.md`, `CHANGELOG.md`, `nextflow.config`, `conf/base.config`, `subworkflows/local/utils_nfcore_*_pipeline/main.nf`, `.gitignore` | **Per-hunk decision (see below)** |

There is no "pure take TEMPLATE" bucket. Even files that look like template scaffolding routinely carry pipeline-specific extensions: CI workflows often have custom jobs (private-data download, licensed-tool tests, megatests), `.nf-core.yml` carries pipeline-specific lint ignores, and `nf-test.config` may pin a custom testdata branch.

**When unsure on any individual hunk, stop and prompt the user.** Paste the conflict hunk verbatim, say what each side does in one sentence, and offer the candidate resolutions (take TEMPLATE / keep ours / hybrid with specifics). Template merges are judgment calls — silently guessing wrong is harder to recover from than asking.

### Hybrid resolution rules

- **CI workflows (`.github/workflows/*`)**: take TEMPLATE's structural changes (action pinned SHAs, runner image bumps, NFT_VER), preserve every custom job (e.g. `netmhcpan`, `awsmegatests`, anything keyed off `secrets.*` your pipeline owns). Diff against `upstream/dev` BEFORE the merge to know what's custom: `git diff upstream/TEMPLATE...upstream/dev -- .github/workflows/`.
- **`.nf-core.yml`**: take TEMPLATE's `nf_core_version` bump, keep the pipeline's `lint:` block verbatim.
- **`nf-test.config` / `tests/nextflow.config`**: check whether your pipeline's test configs depend on the existing `pipelines_testdata_base_path` shape — a template-side rewrite can produce double path segments.
- **Version bumps and pinned SHAs** (`nextflowVersion`, action `@SHA`): take TEMPLATE.
- **Doc URLs and rewritten warning text** (`nf-co.re/docs/...`): take TEMPLATE.
- **Numeric tuning the pipeline deliberately diverged on** (`maxRetries`, custom resource labels): keep ours.
- **Workflow `take:` signatures**: if `main.nf` already calls the workflow with the template's new parameter list (multiqc_config, multiqc_logo, etc.), the workflow MUST adopt the new signature even if the body is ours. Verify by reading `main.nf` after auto-merge.
- **README content lists, CHANGELOG version headers**: keep ours — TEMPLATE side is usually `<!-- TODO nf-core: fill in -->` placeholders.
- **`.gitignore`**: align to the **template** (not whatever the pipeline previously had). Don't add `.gitignore` to `.nf-core.yml`'s lint ignore list.

### Auto-merged ≠ correct

Files git resolves without conflict markers can still be broken. After the merge, grep for:

- References to a `take:` parameter whose name was renamed in the auto-merge (classic: template injects `if (monochrome_logs)` but `take:` declared `_monochrome_logs`)
- Duplicate process invocations (e.g. two `MULTIQC(...)` calls — one new-style with tuple input, one old-style with positional channels)

Run `<cmd_prefix> nextflow lint .` immediately after resolving conflicts — it catches these.

## Pitfalls That Silently Break the Merge

| Pitfall | What happens | What to do |
|---|---|---|
| `git stash` mid-merge | Stashes the resolved tree AND clears `MERGE_HEAD`. Branch resets to pre-merge state. `stash pop` brings back changes but **not** the merge relationship | Commit untracked work to a side branch before starting; if it happens, restore with `git update-ref MERGE_HEAD upstream/TEMPLATE` then commit |
| `git add -A` | Stages local dev files (`logs/`, `samplesheets/`, `scripts/`, `run_epp.sh`, `work/`, large untracked tarballs) into the merge commit | Add specific paths or stage only modified files: `git add -u` |
| Skipping `nextflow lint .` | Auto-merged undefined refs slip through and break CI | Always run `nextflow lint .` AND `nf-core pipelines lint` before committing the merge |
| Trusting every local lint failure | User-level `~/.nextflow/config` can inject params that show up as `schema_params: Param X from nextflow config not found in nextflow_schema.json` | Cross-check against `git show upstream/dev:nextflow.config` or wait for CI — if it doesn't reproduce there, it's a local-env false positive |

## Post-Merge Validation Gate

Run, in order, before committing the merge:

```bash
grep -rn "<<<<<<<\|>>>>>>>" --include="*.nf" --include="*.config" --include="*.yml" --include="*.json" --include="*.md" .   # zero hits
<cmd_prefix> nextflow lint .                  # zero errors (warnings ok)
<cmd_prefix> nf-core pipelines lint           # known failures only (e.g. expected .gitignore drift if you chose not to align)
<cmd_prefix> nf-test test tests/<fast>.nf.test --profile=+docker   # smoke-test a quick profile (peptides/proteins/single-end, not full variants)
```

Update `CHANGELOG.md` — bump the `nf-core` row in the active version's `### Dependencies` table:

```diff
- | `nf-core`  | 3.5.2       | 3.5.2       |
+ | `nf-core`  | 3.5.2       | 4.0.2       |
```

## Committing the Merge

```bash
git add <specific-paths>                       # never -A in this branch
git commit -m "Merge nf-core template X.Y.Z into dev"
git push -u origin merge-template-X.Y.Z
```

Open a draft PR against the **`dev`** branch of the upstream repo using the repo's `.github/PULL_REQUEST_TEMPLATE.md` as the body. Keep the description to one sentence — reviewers expect the template structure, not a custom summary.

---

## When NOT to Use This Skill

- Bumping a single nf-core module (use `nf-core modules update`, not a template merge)
- Bumping `nf-core/tools` itself in the dev environment (no branch merge needed)
- Recreating a broken TEMPLATE branch — see Template Sync in the parent `pipelines` skill
